## Challenges & Growth Areas

### Primary lesson — interaction boundaries

**Boundary creation is not a binary label. A story can begin looking like pattern-application and reveal hidden invariants once it starts stitching together real user flow.** Epic 1 gave us "new module = boundary." Epic 2 taught that "new contract between shipped pieces" is equally dangerous and harder to see during prep. Story 2.3 is the archetype: on paper a wizard extension, in practice it forced decisions about template-copy vs schema-write, pre-existing mode state from `setup.bat`, zero-app invariants, exit gating, and cross-story behavior.

**The real signal is not whether a story introduces a new module, but whether it creates a new interaction contract between existing pieces.**

### Evidence patterns

**Pattern 1 — Story 2.2 was a hidden interaction-boundary story.**
Classified as pattern-application. In reality: first interactive TTY prompt flow, first Anthropic SDK integration, first `settings.yaml` writer, first UX error-translation contract. 8 HIGH patches — tied with 2.4 for worst in the epic. Dev's Completion Notes claimed "No implementation surprises"; 23 patches were needed. Calibration gap.

**Pattern 2 — Story 2.4's zero-modes lockout was pure interaction-boundary.**
Main flow wrote `setup_complete` unconditionally. `run_mode_wizard_step` returned early on non-interactive / `KeyboardInterrupt` paths without running its AC #11 exit gate. Happy path: fine. Degraded path: marker lands with zero modes → fast-path locks user out on every subsequent run. **Caught by external audit, not in-skill review.**

**Pattern 3 — Tests that mock away the invariant they are supposed to prove.**
- Story 2.2 integration test bypassed `main()` (second-pass added `TestFullWiringThroughMain`).
- Story 2.5 notice-ordering test patched `_configure_file_logging` to a no-op, so the "Phase A is gone at write time" invariant was never exercised.
- Story 2.3 third pass: **6 claimed fixes had no regression test.**
- Story 2.4 AST guard silently passed relative imports (`node.level != 0` filter).

**Pattern 4 — Degraded paths held the worst defects.**
- 2.3 gate's "decline escape" violated AC #11.
- 2.4 zero-modes lockout violated the spirit of AC #3.
- 2.3 modify-template seeded from shipped defaults instead of the user's current file (data loss on any user who ran setup, edited a mode, and re-ran setup choosing "modify").
- 2.2 retry counter consumed attempts on empty input.
Every one: happy path works, degraded path creates a trapped or data-losing state.

**Pattern 5 — UX voice drift re-surfaces.**
- 2.1: setup.bat used `OK`/`ERROR` text instead of mandated `✓`/`✗` (HIGH).
- 2.4: State A body diverged from AC #1 verbatim copy (HIGH).
Each caught individually; the recurrence signals a prep-time checklist gap.

### Previous-retro echoes

- **Same-session LLM review has independence bias** (Epic 1 observation, still true). Story 2.4's external audit catching the zero-modes lockout is confirmatory evidence. A3 REDESIGN targets this with explicit delta capture on Story 3.5.
- **Test-harness edge cases continue to accumulate.** Integration tests that mock away invariants, AST guards with narrow matchers, fixtures that differ between branches of the same test. Small, but grow if ignored.

## Key Insights

1. **The classification is a planning tool, not a guarantee.** A story labeled "pattern-application" can silently become an interaction-boundary story once implementation begins. The right response is not better labels — it is a short set of questions (A6) that surface interaction boundaries at prep time.

2. **Degraded-path bugs dominate the HIGH-severity patch list.** Happy-path test coverage was uniformly excellent. Degraded-path coverage was the real gap. A9 makes degraded-path tests a prep-time obligation for boundary and interaction-boundary stories.

3. **Review patches without regression tests are a mulligan, not a design.** Story 2.3's third-pass closure existed because first-pass patches shipped without accompanying tests. A7 closes the loop.

4. **Cross-story state is the dominant hidden invariant.** Story 2.4's seam with 2.1's pre-copy, 2.3's modify behavior depending on whether a user had hand-edited after 2.1, 2.5's `_AlwaysHealthyCheck` smoke test that will trip on Story 3.5's Nerve wiring — these are the invariants that story-local thinking cannot see. A10 requires them to be listed in the story spec.

5. **Implementation-confidence vs review-outcome mismatch is a useful signal.** When Completion Notes claim low surprise but review surfaces major missed invariants, the delta is informative. A8 captures this qualitatively (not by patch-count threshold) to keep calibration tight without gaming the Completion Notes wording.

## Action Items

### Carried forward from Epic 1

**A1 — Boundary-first invariant sweep (EXTENDED to interaction boundaries).**
When a story introduces *either* a new integration boundary (new module, new port impl) *or* a new interaction contract between existing pieces, its spec must include explicit sections for lifecycle, teardown under partial failure, concurrency/cancellation, error-translation boundary, test-determinism hooks, and a Review Focus subsection. The interaction-boundary extension is the key Epic 2 refinement.
**Owner:** Sayuj (SM hat, during story prep).
**Applies to:** Any Epic 3+ story where A6's three questions surface a "yes."
**Success criteria:** Zero HIGH-severity findings rooted in missed interaction-boundary invariants.

Format: run the standard same-session review; run a second review in a fresh session with no implementation context; document both finding-sets side by side; write a short decision note on whether the extra cost is worth the signal.
**Owner:** Sayuj.
**Target:** Story 3.5 review stage.
**Success criteria:** A single retro sub-document named `fresh-session-review-3.5-<date>.md` with both finding-sets and a keep/drop recommendation.

### New from Epic 2

**A6 — Interaction-boundary detection questions at story prep.**
For every Epic 3+ story, the prep note must answer three questions:
1. Does this story create a new contract between existing pieces (not just a new module)?
2. Does it define new invariants in the degraded, partial-failure, or cancelled path?
3. Does its behavior depend on prior-story state (files already written, tests already locking behavior, markers already set)?
Any "yes" promotes the story to boundary-sweep treatment (A1) regardless of whether it introduces a new module.
**Owner:** Sayuj.
**Applies to:** Every Epic 3 story, starting with 3.1.
**Success criteria:** Three questions answered in every story file under a new `## Story-type classification` section.

**A7 — Patch + regression test land together.**
Any review patch that fixes a real defect or contract mismatch must include a regression test in the same diff. Pure documentation or story-text cleanup patches are exempt. A patch that ships without a regression test for a real fix is a review-process failure and must be corrected before the story exits review.
**Owner:** Sayuj (review stage).
**Applies to:** All Epic 3+ reviews.
**Success criteria:** No third-pass verification-gap closure needed on any Epic 3 story.

**A8 — Dev self-calibration note (qualitative trigger).**
When implementation notes claim low surprise or low risk, and review later finds major missed invariants, add a short "what I missed during implementation" bullet to the story's Completion Notes. This is a qualitative signal — not gated by patch count. Purpose is to keep the dev-implementation vs review-outcome calibration loop visible.
**Owner:** Sayuj (as dev).
**Applies to:** Any Epic 3+ story where the review surfaces contract-level findings that weren't anticipated in Dev Notes.
**Success criteria:** Self-calibration bullets appear when warranted, not performatively on every story.

**A9 — Degraded-path proof obligation for boundary and interaction-boundary stories.**
For any story promoted to boundary-sweep treatment, the spec must require at least one explicit test for each of:
1. **Happy path** — the primary success case.
2. **Degraded / partial-failure path** — at least one failure mode tested end-to-end (not just mocked at the SUT boundary). *Anti-pattern example from Epic 2: Story 2.5's notice-ordering test patched `_configure_file_logging` to a no-op, so the "Phase A is gone at write time" invariant was never actually exercised — the test proved the code path ran, not the contract it was supposed to enforce.*
3. **Retry / rerun / idempotency path** — the story's behavior when invoked a second time, or after an interrupted prior run.

Every Epic 2 HIGH-severity bug was a category 2 or 3 bug; making these tests a prep-time obligation closes the gap.
**Owner:** Sayuj (during story prep).
**Applies to:** Every boundary / interaction-boundary story in Epic 3+.
**Success criteria:** All three test categories appear in the Tasks/Subtasks list of every promoted story.

**A10 — Cross-story state assumptions section in interaction-boundary stories.**
Any story promoted under A1 must include a `## Depends on prior-story state` section listing explicit assumptions, for example:
- `setup.bat` may already have pre-copied defaults to `%LOCALAPPDATA%/nova/modes/`.
- `settings.yaml` may exist but be incomplete (missing `api_key`).
- `audit_log` row with `action_type = "setup_complete"` may or may not exist.
- Prior tests may mock helper X and therefore bypass invariant Y.

This surfaces the invariants that story-local thinking cannot see. Two of Epic 2's data-loss bugs (2.3 modify-template seeding, 2.4 zero-modes lockout) would have been caught at prep time by this section.
**Owner:** Sayuj (during story prep).
**Applies to:** Every interaction-boundary story in Epic 3+.
**Success criteria:** The section exists in every promoted story and is cited at review time to check for missed invariants.

## Closing Note

Epic 1 taught the team that pattern-creation has an unavoidable first-through-boundary tax. Epic 2 taught that tax also applies to contracts between already-shipped pieces — and that this tax is harder to see, because there is no new file to point at during prep.

The process refinements from this retrospective (A6 detection questions, A9 degraded-path proof, A10 prior-state assumptions) make the tax visible at prep time instead of surfacing it as HIGH-severity review patches and third-pass verification-gap closures.

Epic 2 shipped clean, with zero production incidents, a stable CI gate, and 100% story completion. The carried-forward lesson is not "do less review" — the review layers earned their cost. The refinement is "move the interaction-boundary discovery upstream into story prep so the review layers spend their energy on genuinely-subtle defects instead of invariants that should have been written into the spec."
