## Successes & Strengths

1. **Pattern-setting stories paid off early.** The two-function clock pattern (`_utc_now_iso` module indirection) established in Story 1.3 was reused by Story 1.8 and locked by tests. Once a pattern shipped, later stories copied it cleanly.
2. **AST-based architectural guardrails.** Import-isolation and field-discipline tests introduced in Story 1.2 extended cleanly into 1.4, 1.6, 1.9, 1.10. Architectural drift caught mechanically, not by review vigilance.
3. **Frozen dataclasses + single-worker executor** eliminated entire classes of concurrency bugs by construction.
4. **Adaptive error-handling rules (Story 1.6):** mode files skip-on-error (one bad file doesn't brick the app), singletons hard-fail (corrupted settings = fatal). Operator-aligned behavior.
5. **Adversarial review passes caught real production-fire bugs:** 1.4 transaction dispatch race, 1.5 WAL busy-flag check, 1.10 argparse `KeyboardInterrupt` phase mismatch. These would all have shipped without the review layer.
6. **Deferral discipline.** 6 items deferred, 0 silently dropped. Every deferral documented with target-story or target-epic.
7. **Pattern-application stories got cleaner through the epic.** 1.6/1.7/1.8 had near-zero post-review patches, demonstrating the compounding value of established patterns.

## Challenges & Growth Areas

### Refined diagnosis (Sayuj's framing)

**High-patch stories were not generic "spec ambiguity" problems. They were *integration-boundary + new failure model* problems.** The first story through a new boundary pays a tax for crystallizing hidden invariants around lifecycle, teardown, transactionality, ordering, cancellation, logging, and testability. Once a boundary has a reference story, later stories touching that boundary are cheap.

Boundary-creation stories (high patch cost):
- 1.0 config schema spike — 20 review items
- 1.4 SQLite engine — 13 patches
- 1.5 migration runner — 12 patches
- 1.10 composition root + CLI — 17 patches

Pattern-application stories (low patch cost): 1.2, 1.3, 1.6, 1.7, 1.8, 1.9, 1.11.

**Improvement lever:** not "better specs everywhere" but "richer invariant-sweep on the first story through each new boundary."

### Other patterns

- **Same-session LLM review has independence bias.** Reviews share context with implementation. Worth trialing a fresh-session review on one boundary-creation story to measure whether the extra cost buys meaningful extra signal.
- **Cross-cutting patterns were discovered mid-epic.** `transaction()` context manager crystallized during 1.5 and retrofitted to 1.4. The clock pattern emerged in 1.3, got violated in 1.8, fixed in patch. A patterns registry consulted during story prep could have caught some of this up front.
- **Test-harness edge cases accumulated quietly** (timing-dependent rollback test, `sys.modules` pollution in test helper, 3-digit migration filename cap, unguarded `close()` in handler cleanup). Small, but grow if ignored.
- **Coverage threshold (Story 1.11 AC #7) deferred.** Config wired, enforcement parked. Reasonable for T1 baseline, but silent drift is the failure mode.

---

## Key Insights

1. **The cost of boundary-creation is real and largely unavoidable.** The first story through a new integration boundary will surface invariants that weren't visible from the architecture doc. Plan for it; don't try to eliminate it.
2. **Patterns ship once; rework amortizes forever.** Every cross-cutting pattern introduced in Epic 1 reduced friction for subsequent stories. This justifies front-loading pattern work even when a single story doesn't need it.
3. **Correctness-first trade validated.** High review cost bought zero production-fire bugs shipping. The trade is worth keeping — but the goal for Epic 2 is to shift *where* in the workflow that correctness investment happens (upstream into boundary specs, not only downstream in review patches).
4. **Solo-dev + single-LLM context creates an independence gap.** Worth testing fresh-session review as a mitigation, at least on the highest-risk story in Epic 2.
5. **Deferral documented > deferral silent.** The `deferred-work.md` discipline means Epic 2 starts with a known backlog, not unknown debt.

## Action Items

### 1. Boundary-first invariant sweep
**Rule:** When a story introduces a *new* integration boundary (not just applies an existing one), its spec must include explicit sections for:
- Lifecycle (start/stop ordering, partial-init cleanup)
- Teardown under partial failure
- Concurrency model & cancellation behavior
- Error-translation boundary
- Test-determinism hooks (clock indirection, AST guards, etc.)
- **Review focus subsection** listing the likely failure modes for the reviewer

**Owner:** Sayuj (SM hat, during story prep)
**Applies to:** Any Epic 2+ story introducing a new port implementation (first Skin story, first Eyes adapter, first Hands adapter, wizard flows with persistence, etc.)
**Success criteria:**
- Primary: zero high-severity review findings caused by missing boundary invariants
- Stretch: materially reduced total post-review patch volume vs. Epic 1 boundary stories

### 2. Cross-cutting patterns registry (operational, not reference-only)
Create `docs/cross-cutting-patterns.md` listing patterns discovered in Epic 1:
- Two-function clock indirection
- AST import-isolation + field discipline
- Frozen dataclass + single-worker executor
- Error-translation-at-boundary (opaque messages, narrow catches, chain via `from`)
- Per-file skip-on-error vs. singleton hard-fail
- `transaction()` async context manager
- Partial-init cleanup in composition root

**Rule:** Every story that touches a relevant boundary must include a Dev Notes line: `Patterns consulted: <names>`. This makes the registry part of story-prep workflow, not background reference.

**Owner:** Sayuj
**Deadline:** Before Story 2.1 starts
**Success criteria:** Doc exists; Stories 2.1–2.5 cite it in dev notes where relevant.

### 3. Fresh-session review trial
On Story 2.3 (mode-creation wizard), run one review pass in a fresh session with no implementation context. Compare the finding-set against same-session review output. Decide whether the extra cost is worth the independence.

**Target:** Story 2.3 (chosen over 2.4 because mode-creation wizard is a true new-boundary story — input validation, persistence, UX flow, error surfaces).
**Owner:** Sayuj
**Success criteria:** Retro note in Story 2.3 comparing finding-sets across review modes, with a recommendation for Epic 3+.

### 4. Reserved Windows filenames & path validation — promoted to explicit AC in Story 2.1
**Rationale:** First-run setup is where product trust is built or broken. Windows path semantics are the main path for this project, not an edge case. Opaque OS errors at setup time feel like "the product is flaky," not "I typed something wrong."

**Story 2.1 must explicitly reject / validate:**
- Reserved names: `CON`, `NUL`, `AUX`, `COM1–9`, `LPT1–9`, `PRN` (case-insensitive, with/without extension)
- Invalid characters: `<`, `>`, `:`, `"`, `|`, `?`, `*`
- Trailing dots or trailing spaces on any path segment
- Normalize resolved paths; validate after normalization
- Surface a **clear product message** when a path exceeds the current host's supported Windows limits (do NOT hard-code `MAX_PATH = 260` unless long-path support is explicitly in-scope for 2.1)

**Owner:** Sayuj (as story author, then as dev)
**When:** Story 2.1 AC addition before implementation starts
**Priority:** High — user-facing trust

### 5. Coverage threshold decision
By the end of Epic 2: either measure the T1 baseline and set an enforced threshold in CI, **or** explicitly defer to Epic 3 with written rationale in `deferred-work.md`. Silent drift is the only unacceptable outcome.

**Owner:** Sayuj
**Deadline:** End of Epic 2
**Priority:** Medium

## Closing Note

Epic 1 traded speed for correctness in the places where correctness compounds. That trade was the right one for a foundation epic, and the evidence (zero production-fire bugs, stable CI gate, clean mypy strict, compounding pattern reuse) supports keeping the correctness-first discipline in Epic 2.
